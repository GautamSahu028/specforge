## Backend Overview

The backend is split into **two FastAPI services**:

| Service | Port | Purpose |
|---|---|---|
| `backend/` | 3000 | Core REST API (auth, projects, CRUD) |
| `ai-service/` | 8000 | LLM integration (Groq) for parsing & doc generation |

Both use **async Python** (SQLAlchemy 2.0 + aiomysql) backed by **MySQL 8**.

**Overall Architecture:**
```
specforge/
├── backend/                    # FastAPI REST API backend
├── ai-service/                 # FastAPI AI service (Groq LLM integration)
├── frontend/                   # React + Vite frontend
├── docker-compose.yml          # Docker orchestration
├── .env                        # Environment configuration
├── Makefile                    # Development shortcuts
└── README.md                   # Project documentation
```

## Database Schema & Models

**Database:** MySQL 8 (utf8mb4 charset, InnoDB engine)

**Models Location:** `backend/app/models/`

All primary keys are **CUIDs** (25-character custom IDs) generated by `backend/app/models/id_gen.py`.

---

### User (`user.py`)

```
Table: users
- id (String[30], PK) — CUID format
- name (String[255])
- email (String[255], UNIQUE, INDEXED)
- password_hash (String[255])
- created_at (DateTime with timezone)
- updated_at (DateTime with timezone, auto-updated)
```

---

### Project (`project.py`)

```
Table: projects
- id (String[30], PK) — CUID format
- name (String[100])
- description (String[500], nullable)
- source_type (Enum: RAW_CODE, OPENAPI_SPEC)
- source_code (Text) — up to 100KB
- framework (String[50], nullable) — e.g., "express", "fastapi"
- status (Enum: PENDING, PARSING, PARSED, GENERATING, COMPLETED, FAILED)
- created_at (DateTime)
- updated_at (DateTime)
- Indexes: status, created_at
- Relationships: endpoints (1-to-many), documentation_pages (1-to-many)
```

**Status lifecycle:** `PENDING → PARSING → PARSED → GENERATING → COMPLETED | FAILED`

---

### Endpoint (`endpoint.py`)

```
Table: endpoints
- id (String[30], PK)
- project_id (String[30], FK → projects, CASCADE)
- path (String[500]) — e.g., "/api/users"
- method (Enum: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS)
- summary (String[500], nullable)
- description (Text, nullable)
- tag (String[100], nullable) — grouping label
- created_at, updated_at (DateTime)
- Unique Constraint: (project_id, path, method)
- Indexes: project_id, method, tag
- Relationships: parameters (1-to-many), examples (1-to-many)
```

---

### Parameter (`parameter.py`)

```
Table: parameters
- id (String[30], PK)
- endpoint_id (String[30], FK → endpoints, CASCADE)
- name (String[200])
- location (Enum: QUERY, PATH, BODY, HEADER)
- type (String[50]) — e.g., "string", "integer", "object"
- required (Boolean)
- description (String[1000], nullable)
- schema (JSON, nullable) — complex schema definitions
- created_at (DateTime)
- Index: endpoint_id
```

---

### Example (`example.py`)

```
Table: examples
- id (String[30], PK)
- endpoint_id (String[30], FK → endpoints, CASCADE)
- type (Enum: REQUEST, RESPONSE, ERROR)
- title (String[300]) — e.g., "cURL Example"
- language (String[30], nullable) — e.g., "curl", "python", "javascript"
- code (Text) — the actual code/JSON
- status_code (Integer, nullable) — HTTP status for responses
- created_at (DateTime)
- Indexes: endpoint_id, type
```

---

### DocumentationPage (`documentation_page.py`)

```
Table: documentation_pages
- id (String[30], PK)
- project_id (String[30], FK → projects, CASCADE)
- title (String[300])
- slug (String[300]) — URL-friendly identifier
- content (Text) — Markdown documentation
- order (Integer) — sort order
- created_at, updated_at (DateTime)
- Unique Constraint: (project_id, slug)
- Indexes: project_id, order
```

---

### Enums (`enums.py`)

- **SourceType:** RAW_CODE, OPENAPI_SPEC
- **ProjectStatus:** PENDING, PARSING, PARSED, GENERATING, COMPLETED, FAILED
- **HttpMethod:** GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS
- **ParamLocation:** QUERY, PATH, BODY, HEADER

## Authentication & Authorization System

**Location:** `backend/app/services/auth_service.py`

### JWT Implementation

- **Algorithm:** HS256
- **Secret:** `JWT_SECRET` from environment (default: development key)
- **Expiry:** Configurable via `JWT_EXPIRY_DAYS` (default: 7 days)
- **Token Claims:**
  - `sub` (subject): user_id
  - `exp` (expiration): calculated expiry timestamp
  - `iat` (issued at): current timestamp

### Session Management

- **Storage:** HTTPOnly Cookie (not localStorage)
- **Cookie Name:** `access_token`
- **Extraction:** From cookie in GET `/auth/me` endpoint
- **Validation:** `decode_access_token()` verifies JWT signature and expiry

### Password Security

**Location:** `backend/app/crud/user.py`

- **Hashing:** bcrypt with salt
- **Verification:** Constant-time comparison via bcrypt
- **Minimum Length:** 8 characters (validated at schema level)

## Background Tasks & Services

### Project Parsing Service

**Location:** `backend/app/services/project_service.py`

**`parse_api()` Function:**
1. Calls AI service `POST /parse` endpoint
2. AI Service returns parsed endpoints with parameters
3. Maps raw AI response to `EndpointCreate` schemas
4. Batch creates endpoints, parameters (no examples yet)
5. Updates project status: `PARSING → PARSED` (or `FAILED`)
6. Returns created endpoints

**Status Flow:** `PENDING → PARSING → PARSED` (or `FAILED`)

---

### Documentation Generation Service

**Location:** `backend/app/services/project_service.py`

**`generate_docs()` Function:**
1. Validates project is in `PARSED` or `COMPLETED` status
2. Sets status to `GENERATING`
3. For each endpoint:
   - Calls AI service `POST /generate` endpoint
   - AI returns examples (REQUEST, RESPONSE, ERROR) and markdown doc
   - Stores examples in database
   - Creates `DocumentationPage` with markdown content
4. Sets status to `COMPLETED`
5. Returns updated project

**Status Flow:** `PARSED → GENERATING → COMPLETED` (or `FAILED`)

---

### AI Client Service

**Location:** `backend/app/services/ai_client.py`

- **Base URL:** `AI_SERVICE_URL` from env (default: `http://localhost:8000`)
- **Timeout:** 120s (connect: 10s)
- **Retry Logic:** Automatic retry on connection errors and 5xx status codes (max 2 retries)
- **Exponential Backoff:** Delay = attempt × 2 seconds
- **Methods:**
  - `parse_api(source_code, source_type, framework)` → POST `/parse`
  - `generate_docs(endpoint_dict)` → POST `/generate`

## AI Service Architecture

**Location:** `ai-service/`

### Routes (`ai-service/app/routes/api.py`)

| Method | Endpoint | Description | Error Handling |
|--------|----------|-------------|----------------|
| POST | `/parse` | Parse API code/spec | 400 validation, 502 AI failure |
| POST | `/generate` | Generate docs for endpoint | 400 validation, 502 AI failure |

---

### Parse Service (`ai-service/app/services/parse_service.py`)

**Input:** `ParseRequest` with source_code, source_type, framework

**Process:**
1. Builds user prompt from template with source code
2. Calls LLM with system and user prompts
3. Expects JSON response with `endpoints` array
4. Sanitizes each endpoint:
   - Validates method (GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS)
   - Validates path starts with "/"
   - Validates parameter locations (QUERY, PATH, BODY, HEADER)
   - Defaults invalid locations to QUERY
5. Filters out invalid endpoints
6. Returns parsed endpoints with framework_detected

**Output:** `ParseResponse` with array of `ParsedEndpoint` objects

---

### Generate Service (`ai-service/app/services/generate_service.py`)

**Input:** `GenerateRequest` with endpoint details (path, method, summary, description, parameters)

**Process:**
1. Serializes parameters to JSON
2. Builds user prompt from template
3. Calls LLM expecting JSON with `examples` and `documentation`
4. Sanitizes examples:
   - Validates type (REQUEST, RESPONSE, ERROR)
   - Preserves language and code
5. Extracts documentation (title and Markdown content)

**Output:** `GenerateResponse` with examples array and documentation object

---

### LLM Service (`ai-service/app/services/llm_service.py`)

**Provider:** Groq
- **Model:** Configurable (default: llama-3.1-70b-versatile)
- **Max Tokens:** 4096 (configurable)
- **Temperature:** 0.1 (low randomness for consistency)
- **Response Format:** JSON object

**Retry Logic (tenacity):**
- Max 3 attempts
- Backoff: exponential (min 2s, max 10s)
- Retries on: ConnectionError, TimeoutError
- Logs warnings before retry

**JSON Parsing:**
- Strips markdown fences (``` markers)
- Validates JSON structure
- Raises ValueError on decode failure

---

### Prompt Templates (`ai-service/app/prompts/templates.py`)

**PARSE_SYSTEM_PROMPT:**
- Instructs to extract ALL endpoints
- Validates method, location, type enums
- Infers types from code context
- Groups by tag
- Handles both raw code and OpenAPI specs

**PARSE_USER_TEMPLATE:**
- Takes source code and framework hint
- Requests JSON-only response

**GENERATE_SYSTEM_PROMPT:**
- Instructs to generate comprehensive documentation
- Expects REQUEST examples (curl, Python, JavaScript)
- Expects RESPONSE examples (successful + errors)
- Requests realistic data and valid Markdown

**GENERATE_USER_TEMPLATE:**
- Takes method, path, summary, description, parameters (JSON)
- Requests JSON-only response

## Complete API Flow Examples

### User Registration & Authentication Flow

```
1. POST /api/auth/register
   Request:  { name, email, password }
   Response: { user: { id, name, email, created_at }, message }
   Side Effect: Sets HTTPOnly cookie with JWT

2. GET /api/auth/me
   Reads:    access_token cookie
   Response: { id, name, email, created_at }

3. POST /api/auth/logout
   Clears:   access_token cookie
```

### Project Creation & Documentation Flow

```
1. POST /api/projects
   Input:  { name, description, source_type, source_code, framework }
   Output: { id, status: PENDING, ... }
   Storage: Stores source_code as-is

2. POST /api/projects/parse-api
   Input:  { projectId }
   Steps:
     a. Set status: PARSING
     b. Call AI Service: POST /parse
     c. AI returns endpoints array
     d. Batch create Endpoint, Parameter rows
     e. Set status: PARSED
   Output: Array of created endpoints

3. POST /api/projects/generate-docs
   Input:  { projectId }
   Steps:
     a. Check status is PARSED or COMPLETED
     b. Set status: GENERATING
     c. For each endpoint:
        - Call AI Service: POST /generate
        - Create Example rows (REQUEST, RESPONSE, ERROR)
        - Create DocumentationPage row
     d. Set status: COMPLETED
   Output: Updated project with populated examples and doc pages

4. GET /api/projects/{projectId}
   Output: Full project with nested endpoints, parameters, examples, doc pages

5. GET /api/export/{projectId}/markdown
   Output: text/markdown file with formatted documentation
```

### Search & Discovery Flow

```
GET /api/search?q=users&projectId=xyz&page=1&limit=20
- Searches endpoints.path, summary, description, tag (ILIKE)
- Filters by project_id if provided
- Returns paginated results with endpoint details
```

### `useAsync(asyncFn, immediate = false, deps = [])`

- **Returns:** `{ data, error, loading, execute, reset }`
- **Purpose:** Generic async operation state manager — eliminates boilerplate loading/error/data patterns
- **Features:**
  - `execute(...args)`: Manually trigger the async function with arguments
  - `reset()`: Clear all state back to initial values
  - `immediate = true`: Auto-runs `asyncFn` on mount
  - Tracks mounted status via ref to prevent state updates after component unmounts
  - Re-runs when `deps` change (like useEffect dependencies)

---

### `useDebounce(value, delay = 300)`

- **Returns:** Debounced value (updates only after `delay` ms of no change)
- **Purpose:** Delays expensive operations (API calls) triggered by fast input changes
- **Used In:** SearchBar (300ms delay), CommandPalette (250ms delay)

**Wraps:** Entire application (in `main.jsx`)

```javascript
useEffect(() => {
  api.getCurrentUser()
    .then(data => setUser(data))
    .catch(() => setUser(null))
    .finally(() => setIsLoading(false));

**Persistence Mechanism:**
- Backend sets HttpOnly cookie on login/register

- On refresh: `GET /auth/me` re-hydrates user from valid cookie

# HOW FRONTEND COMMUNICATES WITH BACKEND

---

## The Axios Instance (`frontend/src/services/api.js`)

This is the **single source of truth** for all backend communication. Every API call in the app goes through this configured instance.

```
Base URL:     /api
Timeout:      180,000ms (3 minutes — long enough for LLM-backed calls)
Credentials:  true (automatically sends HttpOnly cookies on every request)
Content-Type: application/json
```

---

## The Vite Proxy (Development)

The browser never directly contacts `localhost:3000`. Instead, the frontend hits `localhost:5173/api/...` and Vite's dev server transparently proxies the request:

```
Browser → localhost:5173/api/projects
              ↓  (Vite proxy)
          localhost:3000/api/projects  (FastAPI backend)
```

In Docker, the proxy points to `http://backend:3000` (the Docker service name). This also sidesteps CORS entirely in development.

---

## Response Interceptor — Envelope Unwrapping

All backend responses are wrapped in `{ success: true, data: ... }`. The interceptor automatically unwraps this so component code receives just `data`:

```
API response:  { success: true, data: { id: "...", name: "..." } }
What component gets: { id: "...", name: "..." }
```

**Error Enhancement:**
```
error.response.data.error.message  →  err.message
error.response.data.error.code     →  err.code
error.response.status              →  err.status
```

This means all `catch (err)` blocks across the app get consistent, enriched error objects sourced from the backend's unified error format.

## Authentication Flow in Detail

```
1. User fills login form → LoginPage calls api.login(email, password)
   ↓
2. POST /auth/login hits FastAPI backend
   Backend verifies credentials, creates JWT
   Backend responds: Set-Cookie: access_token=<jwt>; HttpOnly; SameSite=Lax
   ↓
3. Browser stores cookie (HttpOnly = JS cannot read it)
   ↓
4. Axios withCredentials: true sends this cookie on EVERY subsequent request automatically
   ↓
5. On every app load (refresh/revisit):
   AuthContext useEffect fires → GET /auth/me
   Backend reads cookie, verifies JWT, returns user object
   setUser(data) → isAuthenticated = true → app unlocks
   ↓
6. On logout: POST /auth/logout
   Backend clears cookie with Set-Cookie: access_token=; Max-Age=0
   Frontend: setUser(null) → isAuthenticated = false → redirected to /login
```

**Security properties:**
- JWT never visible to JavaScript (HttpOnly cookie)
- No localStorage — immune to XSS token theft
- SameSite: Lax — protects against CSRF

## New Project Creation — 3-Step Sequential Flow

`NewProjectPage.jsx` chains three API calls in sequence, blocking each until the previous completes:

```
User clicks "Generate Documentation"
  ↓
Step 0: POST /projects
  Payload: { name, source_type, source_code, framework? }
  Backend: Creates DB record (status: PENDING)
  Response: { id, name, status: "PENDING", ... }
  UI: Shows "Creating project..."
  ↓
Step 1: POST /projects/parse-api
  Payload: { projectId: id from step 0 }
  Backend:
    - Sets status → PARSING
    - Calls ai-service POST /parse with source_code
    - AI extracts all endpoints and parameters
    - Batch-inserts Endpoint + Parameter rows
    - Sets status → PARSED
  UI: Shows "Parsing endpoints..."
  ↓
Step 2: POST /projects/generate-docs
  Payload: { projectId: id }
  Backend:
    - Sets status → GENERATING
    - For EACH endpoint:
        Calls ai-service POST /generate
        AI returns examples + markdown doc
        Inserts Example rows
        Inserts DocumentationPage row
    - Sets status → COMPLETED
  UI: Shows "Generating docs..."
  ↓
navigate(`/projects/${id}`)
```

The entire pipeline can take **30–90 seconds** depending on how many endpoints the project has (one LLM call per endpoint in step 2). The `ProcessingOverlay` shows which step is currently active.

## Search Architecture

### SearchBar (project-scoped, in ProjectViewPage top bar)

```
User types in search field
  ↓
useDebounce (300ms delay)
  ↓  (if query.length >= 2)
GET /search?q={query}&projectId={id}&limit=10
  ↓
Results shown in dropdown
  ↓
User clicks result → onSelect(endpoint) → sidebar highlights + EndpointPage loads
```

### CommandPalette (global, Cmd+K)

```
User presses Cmd+K → palette opens
User types query
  ↓
useDebounce (250ms delay)
  ↓  (two parallel requests)
  ├── GET /search?q={query}&limit=8    (server-side endpoint search)
  └── GET /projects?limit=50           (all projects, filtered client-side)
  ↓
Results shown in two sections: Projects | Endpoints
  ↓
Endpoint click → navigate('/projects/{ep.project_id}', { state: { activeEndpointId: ep.id } })
Project click  → navigate('/projects/{proj.id}')
```

The `activeEndpointId` in navigation state is read by `ProjectViewPage` to automatically highlight and show the endpoint that was clicked from the CommandPalette.

## Export Architecture

|---|---|---|

| HTML | **Client-side** (JavaScript string template) | Creates full HTML with embedded CSS → Blob → `<a download>` click |

## TryItPanel — Direct Browser Fetch (Not Proxied)

The Try It feature makes **real HTTP requests directly from the browser** using the native `fetch()` API — NOT through the Vite proxy or the backend.

```
User sets base URL (e.g., http://localhost:3000)
User fills path params, request body, headers
Clicks Send
  ↓
TryItPanel builds:
  URL: baseURL + path (with {param} substitutions)
  Headers: from header form fields
  Body: JSON string (for POST/PUT/PATCH only)
  ↓
fetch(url, { method, headers, body })
  ↓
Parses response as JSON (falls back to text)
Shows: status code, elapsed time, response headers, formatted body
```

**Important:** This means the API server being tested must either:
- Be running on localhost with CORS open to `localhost:5173`, OR
- Be a server the browser can reach directly

Failures show diagnostic messages: "Check if your server is running on the correct port" / "CORS may be blocking this request"

## Error Handling Pattern (Universal)

Every API call in every component follows this identical pattern:

```javascript
try {
  const data = await someApiFunction(args)
  setState(data)
  toast.success("Operation succeeded!")
} catch (err) {
  toast.error(err.message || "Something went wrong")
  // optionally: setError(err.message) for inline ErrorState display
}
```

The axios interceptor ensures `err.message` always contains a human-readable string sourced from the backend's `error.message` field (from the unified `{ success: false, error: { code, message } }` format). Components never need to parse error responses themselves.

## Data Flow Summary (End to End)

```
Browser (React SPA)
  └── api.js (axios, baseURL="/api", withCredentials=true)
        └── Vite Dev Proxy (/api → localhost:3000)
              └── FastAPI Backend (port 3000)
                    ├── MySQL Database (port 3306)
                    └── AI Service HTTP Client
                          └── FastAPI AI Service (port 8000)
                                └── Groq LLM API (external, HTTPS)
```

Every user action that modifies data (create project, parse, generate, rename, delete) flows left to right through this chain. Read operations (list projects, get project, search) stop at the MySQL database. AI-heavy operations (parse-api, generate-docs) continue all the way to Groq and back.
